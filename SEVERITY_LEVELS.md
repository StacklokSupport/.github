# Severity Levels

## Overview

When creating a GitHub issue, please select the appropriate severity level so we can respond according to our SLA. Choose the severity that best matches the impact on your operations.

---

## Severity 1: Critical Production Issue
**Production system outage, complete inability to provide service**

### Description
Your whole production system is down and you are completely unable to provide your services. A critical part of your service, data, or infrastructure is entirely inaccessible to you or your end users.

### Examples
- Complete system outage
- Production database unavailable
- All users unable to access the system
- Critical infrastructure failure preventing any work

### Response Commitment
- **Response Time**: 1 hour
- **Support Hours**: Monday-Friday, 6 AM - 6 PM Pacific Time
- **Collaboration**: We'll invite you to Slack immediately for real-time resolution
- **What to Include in Your GitHub Issue**:
  - Select "Severity 1" in the issue template
  - Clear statement: "Production is completely down"
  - What is down
  - When it started
  - Number of users/services affected
  - Any error messages
  - What you've already tried

---

## Severity 2: High Priority Issue
**Operations can continue in a restricted fashion, but productivity affected**

### Description
Your operations can continue in your production or staging environments, but productivity is affected. Your environment is operational but significantly degraded in some respect.

### Examples
- Major feature not working but system still accessible
- Significant performance degradation
- Workarounds exist but are time-consuming
- Issue affecting multiple users or important workflows
- System operational but with reduced functionality

### Response Commitment
- **Response Time**: 4 hours
- **Support Hours**: Monday-Friday, 6 AM - 6 PM Pacific Time
- **Collaboration**: We may use Slack for faster troubleshooting
- **What to Include in Your GitHub Issue**:
  - Select "Severity 2" in the issue template
  - What functionality is impaired
  - How productivity is affected
  - Number of users impacted
  - Current workarounds being used
  - Error messages or screenshots

---

## Severity 3: Standard Issue
**Partial, non-critical loss of functionality**

### Description
Partial non-critical loss of functionality. Minor problems that do not affect the stability of your infrastructure. Technical questions, configuration issues, or issues that affect a small number of end users.

### Examples
- Minor bugs with workarounds
- Configuration questions
- Non-critical feature issues
- Issues affecting few users
- Documentation requests
- General technical questions

### Response Commitment
- **Response Time**: 1 business day
- **Support Hours**: Monday-Friday, 6 AM - 6 PM Pacific Time
- **Collaboration**: Most Severity 3 issues are resolved directly in GitHub
- **What to Include in Your GitHub Issue**:
  - Select "Severity 3" in the issue template
  - Description of the issue
  - Any workarounds you've found
  - Steps to reproduce (if applicable)
  - Expected vs actual behavior

---

## Severity 4: Low Priority / General Inquiries
**Feature requests, 'how do I' requests, non-severe issues**

### Description
Non-severe issues, feedback and suggestions, advice on best practices, feature requests, or missing/incorrect documentation.

### Examples
- Feature requests
- Best practice questions
- Documentation improvements
- General "how do I..." questions
- Product feedback
- Non-urgent suggestions

### Response Commitment
- **Response Time**: No specific SLA
- **Support Hours**: Monday-Friday, 6 AM - 6 PM Pacific Time
- We'll respond as time permits, typically within a few business days

---

## Important Notes

### Unsupported Systems
MCP servers not built and maintained by ToolHive are considered "Unsupported Systems." Support for these is provided on a reasonable-efforts basis with no warranties or guarantees.

**Questions about severity?** Create a Severity 4 issue and ask - we're happy to help you assess the right level!
